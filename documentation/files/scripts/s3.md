# Scripts : S3

**Emplacement :** `scripts/s3/`

Scripts de versement, d'inventaire et de suppression des
[fichiers de conservation](../donnees/fichiers.md) sur le stockage S3.

## Pourquoi utiliser un script et non un utilitaire type Cyberduck ?
On verse les fichiers dans le S3 avec deux métadonnées d'identification :
- `uuid`
- `checksum_md5`
Ces deux métadonnées permettent d'identifier sûrement le fichier et de vérifier son intégrité.
Or les utilitaires disponibles (Cyberduck, ...) ne permettent pas l'envoi de ces informations.

---

## Configuration

La connexion est définie dans `scripts/s3/conf.yml` (non versionné, voir le
modèle `conf_empty.yml`) : URL du endpoint, région et identifiants pour trois
profils d'accès :

| Profil | Usage |
|---|---|
| `user_ro` | Lecture seule (profil par défaut) |
| `user_rw` | Lecture/écriture (versement, suppression) |
| `user_test` | Tests |

Le module `rbx_s3.py` encapsule la connexion boto3 et expose deux classes :
- `Rbx_client` : client S3, avec une méthode `upload()` qui verse un fichier
  puis vérifie sa présence et sa taille (`head_object`) ;
- `Rbx_resource` : ressource S3, avec une méthode `delete_object()`.

---

## Versement : `upload.py` / `upload_incommunicable.py`

Versent des lots de fichiers vers le bucket communicable (`upload.py`) ou
incommunicable (`upload_incommunicable.py`).

- **Entrée** : fichiers CSV déposés dans `data/transfert/`, listant les fichiers
  à verser (colonnes `name`, `path`, `uuid`, `checksum_md5`, `size`, `s3_key`).
  Les chemins sont résolus depuis le partage réseau défini par la variable
  `prefix` en tête de script.
- **Traitement** : versement en parallèle (`ThreadPoolExecutor`). Chaque objet
  reçoit les tags S3 `uuid` et `checksum_md5`. Après versement, la taille de
  l'objet est comparée à la taille attendue.
- **Sortie** : un journal CSV par lot dans `result/`
  (`{lot}_upload_{horodatage}.csv`), avec le résultat et l'erreur éventuelle
  de chaque fichier.

---

## Inventaire : `list_all.py`, `list2.py` à `list6.py`

Listent les objets d'un bucket avec leurs métadonnées (clé, date de modification,
taille, classe de stockage) et leurs tags `uuid` et `checksum_md5`.

- `list_all.py` : parcourt tout le bucket communicable, sortie en tranches de
  10 000 objets dans `result/listing-{bucket}_all_{n}.csv.gz`.
- `list2.py` à `list6.py` : variantes ciblées sur un bucket ou des préfixes
  particuliers (ex. `list6.py` liste un préfixe du bucket incommunicable).

Les listings consolidés sont conservés dans `data/s3/s3_listing_{date}.csv.gz`.

---

## Analyse : `repertoires_traitement_s3_ko.py`

Liste les répertoires contenant des fichiers dont le traitement S3 n'est pas
réalisé. Voir [sa page dédiée](repertoires_traitement_s3_ko.md).

---

## Suppression : `delete.py`

Supprime du bucket les objets dont les clés sont listées dans un CSV
(colonne `key`). Le chemin du CSV et le bucket sont définis en tête de script.

**À manier avec précaution** : la suppression est immédiate et sans confirmation.
